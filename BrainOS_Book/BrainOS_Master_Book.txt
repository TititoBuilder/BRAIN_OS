The foundational idea introduced and established in this session is what we

engines serving two completely different cognitive purposes.

ENGINE ONE — PRECISION (ElevenLabs Reader / Local TTS)
Purpose: Consumption. You feed it your technical files and it produces clean,
narrated audio you absorb while moving. Its job is to keep your brain
saturated with the exact content of your own codebase even when you are not
sitting at a screen. It reads precisely what is written. It does not
interpret, synthesize, or connect ideas. It is your learning channel.

ENGINE TWO — SYNTHESIS (NotebookLM)
Purpose: Creation. You feed it structured documents and it generates
AI-powered discussions, cross-document connections, and creative insights
you use to develop new content for your soccer bot. It does not read your
files literally — it thinks about them, finds patterns, and surfaces ideas

Your exact formulation: Engine One makes Cristian learn. Engine Two makes

generation pipeline. This is a self-reinforcing loop — the output of your

SECTION 4 — converter.py: BUILT, TESTED, CONFIRMED WORKING
────────────────────────────────────────────────────────────────────────────────

The first deliverable of this session is converter.py, which we described
as Station One in the content factory. It lives in your project root.

WHAT IT DOES
Accepts any source file in a supported format and produces two output files
in the converted/ subfolder, which it creates automatically if it does not
exist. The two outputs are:

  filenameTTS.txt       — cleaned plain text for TTS narration
  filenameNotebookLM.md — structured Markdown for NotebookLM upload

SUPPORTED FILE FORMATS
  .py    Python source files
  .pdf   PDF documents (uses PyMuPDF primary, pdfplumber fallback)
  .csv   Comma-separated data (uses pandas for summary + sample table)
  .tsv   Tab-separated data (same as CSV)
  .json  JSON files (describes structure in plain English, shows content)
  .docx  Word documents (preserves heading hierarchy)
  .txt   Plain text files
  .md    Markdown files (pass-through, no modification needed)

ARCHITECTURE NOTE — OPEN-CLOSED PRINCIPLE
Each format has its own dedicated converter class. Adding a new format
in the future (example: .ipynb Jupyter notebooks, .html web pages) requires
writing exactly one new class without modifying any existing code. This is
the Open-Closed Principle in practice: open for extension, closed for
modification.

HOW TTS FORMAT WORKS
Strips all Markdown syntax because TTS engines read literally.
  - Code fences → replaced with "Code block begins / Code block ends"
  - Heading markers → replaced with "Section: [heading text]."
  - Bold and italic → markers removed, text preserved
  - Blockquote notes → replaced with "Note: [text]"
  - Tables → removed (tables cannot be narrated meaningfully)

HOW NOTEBOOKLM FORMAT WORKS
Preserves full Markdown structure. Adds a metadata header at the top:
  ---
  Prepared for NotebookLM — Cristian's Dual-Engine Learning System
  Converted: [date and time]
  ---
NotebookLM's Gemini engine reads Markdown as semantic signals. Headings
become navigation anchors. Code blocks are recognized as code. The
structure guides the AI's analysis rather than being treated as noise.

USAGE COMMANDS
  python converter.py soccerbot.py --engine both
  python converter.py matchdata.csv --engine tts
  python converter.py architecture.pdf --engine notebooklm
  python converter.py notes.docx --engine both --output ./converted

LESSON LEARNED DURING TESTING
The first test run produced "File not found: soccerbot.py" because the

which room you mean.

THE CHUNKING ARCHITECTURE — WHY IT MATTERS
Neural TTS models cannot process an entire document as one input. The
solution is to split input text into sentence-level chunks of approximately
400 characters, generate audio for each chunk independently, add a 0.3
second silence between chunks for natural breathing rhythm, then
concatenate all audio arrays into one seamless waveform. The listener
hears one continuous narration with no audible seams.

This split-process-reassemble pattern is the same architectural pattern
your soccer bot already uses when chunking large prompts before sending
to the Claude API. Recognizing this pattern as a fundamental building
block of production AI systems — rather than a quirk of TTS — is the
engineering insight worth carrying forward.

SECTION 8 — THE LANGUAGE QUESTION: WHY BOTH SPANISH AND ENGLISH
────────────────────────────────────────────────────────────────────────────────

You asked: does the language I record in matter for voice model training?

The answer required understanding two separate things a TTS model learns:

ACOUSTIC IDENTITY (language-independent)
Your fundamental frequency, formant patterns, breath placement, timbre —
the physical characteristics that make your voice yours. These are captured
regardless of what language you speak during training. A Spanish speaker
and an English speaker with identical vocal anatomy produce similar acoustic
fingerprints. This is the part that makes the voice recognizably Cristian.

PHONEMIC COVERAGE (language-dependent)
The specific set of sounds your training data contains. English uses
approximately 44 phonemes. Spanish uses approximately 24. Significant
overlap, but not complete. Spanish lacks several sounds English requires
constantly: the "th" sounds, the short "i" versus long "ee" distinction,
and the schwa vowel that appears in almost every unstressed English syllable.

If you record only in Spanish, the model learns your acoustic identity but
has never heard you produce specifically English phonemes. When synthesizing
English text it must extrapolate — the result is your voice with subtly
incorrect English phoneme production. Recognizable but slightly off.

SOLUTION: Record both languages. Use both as training data.
  voicees01.wav through voicees04.wav — Spanish sessions
  voiceen01.wav through voiceen04.wav — English sessions

YOUR BILINGUAL NATURE AS A TECHNICAL ASSET
Spanish is a syllable-timed language — each syllable gets roughly equal
duration. English is stress-timed — stressed syllables are longer and
unstressed syllables get compressed. Native English speakers develop
stress-timing automatically. Spanish-native English speakers often retain
a more syllable-timed cadence, which synthesizes as clearer and more
deliberate narration. Each word receives its full weight rather than
being swallowed in the natural compression of native English speech.
This is not a limitation. It is a distinguishing characteristic.

A model trained on both languages also handles code-switching — the natural
blending of Spanish and English in your thinking and speaking — with
fluency that a monolingual model cannot produce.

sentence end. Pacing slows before the question mark. Breath placement

the full dynamic range of your voice — statement, reflection, inquiry,
emphasis — rather than only the declarative register. The result is
synthesized speech that sounds human rather than mechanical.

SECTION 10 — THE PREDATOR VS DELL ANALOGY
────────────────────────────────────────────────────────────────────────────────

This conceptual framework was developed during the session and is worth
preserving because it captures a real transition in capability level.

DELL WORKFLOW — BOLT-ACTION RIFLE
Reliable. One action at a time. Manual reload between every step.
Find document → open it → read it → take notes → close it → open notes
app → type what you remembered. You were the pipeline. Every unit of
cognitive energy spent moving information from one place to another was
cognitive energy not available for understanding what the information meant.

PREDATOR WORKFLOW — AUTONOMOUS FACTORY
Issue one command. The pipeline identifies the format, processes it two
ways for two purposes, generates audio, and delivers finished outputs
while your attention is free. You are the architect. The machine is the
processing engine. That role reversal is what "agentic workflow" means
in practice, not just in theory.

THE HARDWARE THRESHOLD POINT
The Predator's advantage is not quantitative — it is categorical.
CUDA-accelerated GPU makes local neural network inference practical.
Kokoro-82M on GPU: seconds per chunk.
Kokoro-82M on CPU: minutes per chunk.
The hardware does not just run the workflow faster. It enables a category
of workflow that did not previously exist on your machine.

Welcome to your DaVinci Resolve MCP Server deep dive. This audio guide covers everything you've built, how it works, the terminology behind it, and where it's going next. This isn't a generic tutorial. This is specific to your project, your machine, and your workflow. Let's get into it.

Chapter one. What you built and why it matters.

You built an MCP server that lets Claude control DaVinci Resolve 20 through natural language. Instead of memorizing dozens of keyboard shortcuts, you can now open Claude Desktop and say things like "save my project," "add a marker here," or "cut at the playhead," and Resolve executes the command in real time.

Let's break down what that means. MCP stands for Model Context Protocol. It's an open standard created by Anthropic that lets AI models like Claude communicate with external tools and services. Think of MCP as a universal plug system. On one side, you have Claude, which understands natural language. On the other side, you have DaVinci Resolve, which understands keyboard shortcuts. The MCP server sits in the middle and translates between the two.

This is significant because it turns Resolve from a manual tool into something you can operate conversationally. For your BreakingDown Futbol content workflow, this means you could eventually automate entire editing sequences. Imagine saying "mark all the goal moments in this timeline" or "cut everything between these two markers and export it." That's the direction this project is heading.

Chapter two. The architecture explained.

Your system has three components. Let's walk through each one in detail.

Component one. Claude Desktop. This is the front end. It's the Claude application running on your Windows machine, not the web version at claude dot ai. Claude Desktop has a chat interface where you type natural language. When you type a command related to Resolve, Claude recognizes that a matching MCP tool exists and calls it. Claude Desktop communicates with your MCP server using a transport method called stdio, which stands for standard input output. This means Claude launches your server script as a subprocess and sends messages back and forth through the process's input and output streams. No network ports, no HTTP, just direct process communication.

Component two. The MCP server. This is the bridge. It's a Python script called server dot py that lives at C colon backslash Users backslash titit backslash Projects backslash resolve-mcp-server. The server registers a set of tools, each with a name and a description. When Claude calls a tool, the server receives the request, identifies which tool was called, and executes the corresponding Python function. Each function uses pyautogui to send keyboard shortcuts to the Resolve window.

Component three. DaVinci Resolve. This is the target application. Resolve receives keyboard input from pyautogui exactly as if a human pressed those keys. It doesn't know or care that the input is coming from a Python script. From Resolve's perspective, someone just pressed Control B to make a cut, or Space to toggle playback, or M to add a marker.

Here's the critical insight. The MCP server controls Resolve's software controls only. It does not touch your video files. Your media stays wherever it lives, typically in C colon backslash Media. When the server sends Control S to save, it saves Resolve's project database, which contains your timeline, cuts, markers, and settings. The project database lives at C colon backslash Users backslash titit backslash AppData backslash Roaming backslash Blackmagic Design backslash DaVinci Resolve.

Chapter three. Why pyautogui instead of the native API.

This is an important story because it explains a real-world engineering decision you made.

DaVinci Resolve has a built-in Python scripting API. In older versions like Resolve 18, you could write Python scripts that directly controlled Resolve through a library called fusionscript. You'd import the module, connect to a scripting endpoint, and have full programmatic access to timelines, clips, color grades, and rendering.

Resolve 20 broke this. Specifically, version 20.3.2 removed the external scripting toggle from the user interface entirely. The old API used port 18818 for external connections. Resolve 20 changed to port 20321, but this new port is internal IPC only. IPC stands for Inter-Process Communication. It means the port is designed for Resolve's own internal components to talk to each other, not for external scripts to connect.

You verified this yourself during the build session. The fusionscript DLL loads fine. You can import it without errors. But when you call scriptapp to establish a connection, it returns None. The protocol handshake fails silently. Port 20321 drops external connections immediately with zero bytes returned. The Advanced Settings panel in Resolve 20 is completely empty, with no option to re-enable external scripting.

Your solution was elegant. Instead of fighting with a broken API, you bypassed it entirely. Pyautogui simulates keyboard input at the operating system level. It doesn't need Resolve's permission or cooperation. As long as the Resolve window is focused, pyautogui can send any keystroke. This approach is 100 percent reliable because it operates at a layer below Resolve's application logic.

The tradeoff is that pyautogui can only send commands. It cannot read data back from Resolve. You can tell Resolve to play, cut, or save, but you can't ask Resolve what's currently on the timeline or where the playhead is. For reading timeline data, you'd need the native API, which currently only works in Resolve 18. One option on the horizon is installing Resolve 18.6.6 alongside Resolve 20 specifically for scripting access.

Chapter four. Key terminology.

Let's define every term you need to understand this project.

Term one. MCP, Model Context Protocol. An open standard by Anthropic that defines how AI models communicate with external tools. MCP uses a client-server architecture. Claude is the client. Your server dot py is the server. The protocol defines how to register tools, how to call them, and how to return results.

Term two. Stdio transport. The communication method your MCP server uses. Stdio stands for standard input output. Claude Desktop launches your Python script as a child process. It sends JSON messages to the script's standard input and reads responses from the script's standard output. This is simpler and more reliable than HTTP-based transports because there are no ports to configure, no firewall issues, and no network stack involved.

Term three. Tool registration. When your MCP server starts, it tells Claude what tools are available. Each tool has a name like play underscore pause or cut underscore at underscore playhead, and a description that tells Claude when to use it. Claude reads these descriptions and decides which tool to call based on your natural language request. Good descriptions are critical. If the description says "toggle playback" and you say "play the video," Claude needs to understand those mean the same thing.

Term four. Pyautogui. A Python library for programmatic control of the mouse and keyboard. It works at the operating system level, meaning it can send input to any application. In your project, you use pyautogui dot press for single keys like Space or M, and pyautogui dot hotkey for key combinations like Control B or Control S. Pyautogui doesn't know anything about Resolve. It just sends keystrokes to whatever window is currently focused.

Term five. Pywinauto. A Python library for Windows GUI automation. In your project, you use it specifically for one purpose: finding and focusing the DaVinci Resolve window before sending keystrokes. The ResolveController class uses pywinauto's Desktop class to find any window whose title matches the pattern DaVinci Resolve followed by anything. It then calls set underscore focus to bring that window to the front. Without this step, pyautogui would send keystrokes to whatever window happens to be active, which could be your terminal, your browser, or anything else.

Term six. ResolveController. A Python class you wrote that wraps all the Resolve automation logic. It has a focus method that brings Resolve to the front, and individual methods for each action like play underscore pause, cut underscore at underscore playhead, add underscore marker, save, and others. Each method calls focus first, then sends the appropriate keystroke. This class is the action layer that the MCP server calls into.

Term seven. Claude Desktop config. A JSON file that tells Claude Desktop where your MCP servers live. The file is located at C colon backslash Users backslash titit backslash AppData backslash Roaming backslash Claude backslash claude underscore desktop underscore config dot json. Inside it, you define each MCP server with a name, the path to the Python executable in your venv, and the path to your server script. Claude Desktop reads this file on startup and launches each configured server.

Term eight. IPC, Inter-Process Communication. A mechanism for processes running on the same machine to exchange data. Resolve 20 uses IPC on port 20321 for its internal components. The key distinction is that IPC is designed for trusted internal communication, not for external scripts to connect to. This is why your connection attempts to that port failed.

Term nine. Fusionscript. The DLL, or dynamic link library, that provides Resolve's Python scripting interface. In Resolve 18, you could load fusionscript, call scriptapp, and get a connection object that gave you full control over Resolve. In Resolve 20, the DLL still exists and loads without errors, but the connection handshake fails because the underlying protocol changed.

Term ten. Hammer icon. In Claude Desktop's chat interface, when an MCP server is successfully connected, a small hammer icon appears in the chat input area. This is your visual confirmation that Claude can see and call your MCP tools. If the hammer icon is missing, it means the MCP server failed to connect. The fix is usually to fully quit Claude Desktop and restart it, making sure Resolve is already open before Claude Desktop starts.

Encoding bugs visible only in PowerShell display are not actual bugs. Signal interrupts during Claude Code are environment artifacts. Always verify problems exist in execution context that matters before attempting fixes.

Event-Driven Over Scheduled
Only perform work when something happens (file in incoming\). Scheduled execution wastes resources on empty runs. Event-driven scales to zero cost when idle.

Separate Books Get Separate Files
Cross-project sessions require separate compile outputs per target book. Contamination prevention at file creation beats detection at compile time.

Storage vs Synthesis Distinction
Session transcripts are ingredients, chapters are meals. Synthesis creates refined knowledge that becomes more valuable over time. Storage preserves raw material that becomes harder to search.

Output Tokens Dominate API Cost
At 5x input token price, output generation drives expenses. A verbose chapter costs significantly more than concise output from lengthy source material.

PART ONE — OBSIDIAN ECOSYSTEM
==============================

OBSIDIAN UI OVERVIEW

Obsidian is organized into five main areas. The left ribbon contains
icon shortcuts for files, search, bookmarks, history, graph, and plugins.
The main editor area displays the active note. The right panel shows
backlinks, outgoing links, tags, and properties. The status bar at the
bottom shows word count, cursor position, and vault name.

The most important shortcut in Obsidian is Control plus P, which opens
the Command Palette. Every feature in Obsidian is accessible from here.
Other essential shortcuts include Control plus O for quick open, Control
plus Shift plus F for full-text search, Control plus G for global graph
view, and Control plus Alt plus G for local graph of the current note.

MARKDOWN

Markdown is plain text with simple formatting symbols that render visually.
A hash symbol before text creates a heading. Double asterisks around text
make it bold. A dash before text creates a bullet point. Double square
brackets around a note name create a wiki-link. An exclamation mark before
double square brackets embeds the note inline. Markdown files are universal
except for wiki-links, which are Obsidian-specific. The entire BRAINOS
vault is markdown files, making it fully portable.

YAML METADATA

YAML is the structured data block at the top of every markdown file,
enclosed between triple dashes. It contains properties like tags, creation
date, and version number. Obsidian reads this block to power search,
graph color groups, and filtering. The Properties panel in the right
sidebar lets you edit YAML visually without touching the raw text.

TAGS

Tags are searchable labels that give notes multiple identities beyond
their folder location. A note lives in exactly one folder, but can have
unlimited tags. Typing a hash symbol followed by a word anywhere in a
note automatically registers it as a tag with no setup required.

Tags enable instant filtering in search using the syntax tag colon
hash tagname. They enable graph filtering, where typing a tag query
in the graph filters panel shows only matching nodes. They enable
cross-folder discovery, finding related notes regardless of which folder
they live in. Common tag patterns include hash active, hash archived,
hash broken, hash todo, hash system, and hash protocol.

The key distinction is that folders are physical locations and tags
are logical identities. A session note lives in the 10 CHATS folder
as its home, but can be tagged with hash bdf, hash lancedb, and hash
bug dash fix, making it findable from three different angles.

WIKI-LINKS AND BACKLINKS

A wiki-link uses double square brackets around a note name to create
a connection. Adding a pipe symbol and display text after the note name
creates an aliased link. Adding a hash symbol and heading name links
to a specific section. Backlinks are automatic. When Note A links to
Note B, Note B knows about it and the connection appears in the right
panel. This is how the graph builds itself without any manual work.

GRAPH VIEW

The global graph opens with Control plus G and shows all notes in the
vault as nodes connected by links. The local graph opens with Control
plus Alt plus G and shows only the current note and its direct neighbors.

The graph controls panel has four sections. Filters hide and show notes
by tag, folder, or path. Groups color-code nodes by query, which is how
the twelve-color BRAINOS system works. Display adjusts node size, link
thickness, and text fade threshold. Forces controls how nodes repel and
attract each other.

Node size reflects connection count. The largest nodes are the most
connected and therefore the most important hubs in the knowledge system.
Gray nodes are notes that do not belong to any defined color group,
which indicates they live outside the expected folder structure.

BRAINOS KNOWLEDGE FLOW

The BRAINOS system has two scripts and one protocol. The flow begins
when a session with Claude ends. The first human decision is to run
session close dot py, which is located at C colon backslash BRAIN OS
backslash 09 TOOLS. This script archives the session and saves a
compiled markdown file into the 10 CHATS folder.

The second human decision is to open Claude Code and instruct it to
run the Knowledge Ingestion Protocol version two. Claude Code reads
the protocol document, which lives in 07 SYSTEM, and uses it as
a blueprint to decide what to extract from the session and where to
put it. The protocol is Option C, which means hybrid automated with
review points. Claude Code executes the work automatically but pauses
at complex decisions for human review.

The distinction between a script and a protocol is foundational.
A protocol is a document containing rules and decisions, written in
plain English, that tells Claude Code what to do and why. A script
is executable code that performs specific actions. A script without
a protocol is code with no intent behind it. A protocol without a
script is just documented rules. BRAINOS has both.

The failure point in this flow is human memory. The code works
correctly. The risk is forgetting to trigger the two manual steps
after a session ends.

VS CODE UI LAYOUT

VS Code has five main interface zones. The activity bar on the far
left is a vertical strip of icons for Explorer, Search, Source
Control, Extensions, Debug, and installed extensions like GitLens
and Project Manager. The side panel to the right of it shows the

ESSENTIAL SHORTCUTS

GIT AND GITLENS

VS Code has three Git tools. The built-in Source Control panel shows
changed files, handles staging and commit messages, and pushes to
GitHub. GitLens adds inline blame on every line showing who wrote it,
when, and why. GitLens Inspect provides deep investigation of individual
commits showing exactly what changed.

The key concept is that commits are local saves and push is the upload.
Multiple commits can accumulate locally and be pushed to GitHub in a
single sync operation. This is normal for solo development work.

Commit messages follow conventional commit format. The type prefix is
required and must be one of feat for new features, fix for bug fixes,
docs for documentation changes, chore for maintenance, or refactor for
restructuring. The description follows in lowercase with no period.

Every line in the codebase has a GitLens annotation visible on hover.
The annotation shows the commit hash, author, date, and the full
commit message. This is decision archaeology for code, equivalent to
tracing wiki-links backward in Obsidian.

THE DEBUGGER

The VS Code debugger allows running any Python script with breakpoints.
Setting a breakpoint by clicking in the left margin of any line causes
execution to pause at that point. When paused, the Variables panel on
the left shows every variable in memory with its current value. This
eliminates guessing about what a variable contains during execution.

The debug toolbar has five controls. Continue runs until the next
breakpoint. Step Over executes the current line and moves to the next.
Step Into goes inside the function the current line calls. Step Out
finishes the current function and returns to the caller. Stop ends
the debugging session.

When execution skips lines inside an if block, it means the condition
evaluated to false. The debugger makes this visible in real time,
which is the fastest way to diagnose bugs without adding print statements.

MULTI-CURSOR EDITING

Multi-cursor editing allows editing multiple locations simultaneously.
Control plus D selects the next occurrence of the current word. Pressing

an additional cursor at any position regardless of word matching.

Whatever is typed then replaces all selected occurrences simultaneously.

The F2 key renames a symbol across all files in the project. Unlike
find and replace, F2 understands code context and only renames actual
references to the symbol, not unrelated string matches.
